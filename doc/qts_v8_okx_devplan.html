纵向切片优先
Week 1–3 仅聚焦 OKX ETH 5m 单标的。禁止写多市场适配、禁止接 AI 推断、禁止实现跨截面排名。先跑通 L0→L7 最小闭环再叠加复杂度。每个 Phase 结束必须能独立在 Demo 环境演示，不依赖后续功能。
Rust 热路径 / Python 冷逻辑 — 严格边界

契约先行

Week 2–3
▼
目标
跑通"无 AI"的完整自动化交易闭环：OKX WS → SHM → Rust 特征 → Python Alpha → 门控 → OKX Demo 下单 → 成交回执 → P&L 记录。这是整个系统的骨架，后续所有 Phase 均在此基础上叠加。
任务归属交付物关键说明
SHM 环形缓冲区 Rust core/buffer.rs mmap2 分配共享内存，Arc<RwLock<CircularBuffer>> 并发安全，容量 3600（1s/snapshot）。双缓冲 Ping-Pong 机制防止 Python 读取时 Rust 正在覆写。暴露 DLPack 指针给 Python 侧零拷贝读取。
5m 聚合特征引擎 Rust core/feature_engine.rs 维护两个 SHM 窗口：Tick_Window（实时微观）和 5m_Bar_Window（宏观 Regime）。5m 闭盘前 200ms 利用已有 Tick 预计算 99% 特征，闭盘瞬间仅做 O(1) 修正，立即触发 Alpha 评估（解决"抢跑"问题）。基础 15 维：5m OHLCV、OBI、OFI、Spread_Z、Depth_Imbalance、mom_1m/5m、HV_5m、Hurst。
SHM 零拷贝桥接 Python data/shm_bridge.py torch.from_dlpack 和 np.frombuffer 零拷贝读取 Rust SHM。Python 侧只读，Rust 侧管理生命周期。验证读取延迟 <1μs (p99)。
经验 Alpha 引擎 Python alpha/crypto/obi_v2.py
alpha/crypto/funding_rate_arb.py		 OBI_v2：OBI > 0.6 且 mom_5m > 0，阈值可配置。Funding Rate Arb：定时 REST 轮询 OKX 资金费率，结合 5m 动量判断方向。信号携带 trace_id（与后续 OKX clOrdId 绑定）。
G1/G2 硬门控 + 时间门 Python gating/hard_gating.py G1 流动性：OKX 5m 平均盘口深度 > $50K。G2 Regime：Hurst > 0.5 且 HV_5m < 4%。G5 时间门（OKX 专属）：资金费率结算前 30min 禁止开新仓。所有拒绝信号写 Redis 并异步入 CFL 标签队列。
OKX 执行通道 (Rust) Rust execution/okx_router.rs
execution/order_state.rs		 HMAC-SHA256 签名（ring crate，<5μs），ISO8601 时间戳 + time_offset 补偿（防 50111 错误）。Token Bucket 限流（OKX 20次/2s 限制）。OrderFSM：NEW→POSTED→PARTIAL→FILLED/CANCELED。WS 订阅 orders 频道驱动状态机。BehaviorHumanizer：Rust 层注入 0.1–0.5s Poisson 延迟 + ±0.002 价格微扰。
Harness 基础编排 Python harness/pipeline_v1.py
harness/logging_handler.py		 串联 ShmBridge → Alpha → Gating → Risk(Kelly ≤25%) → Rust Exec。结构化 JSON 日志注入 trace_id，OKX clOrdId 与 trace_id 绑定。Telegram 告警（成交通知、G1/G2 拦截统计、断线预警）。
✓ 验收标准
SHM 读取 <1μs (p99)；特征值与 Python 纯算版余弦相似度 >0.99；OKX Demo 成功完成 100 次"信号→限价单→Fill→P&L"全链路；系统内部延迟（不含 OKX 网络 RTT）<50ms；24h 无内存泄漏
↩ 回滚方案
关闭 feature_engine.rs，恢复 Python 版 feature_engine.py；执行通道切回 ccxt 同步调用；Harness 直接调用 Python 版特征，10s 内完成切换

▼
目标
接入 ResNet 128d 嵌入，构建 50d 经验 ⊕ 128d 深度 → 178d 融合向量。同步启动 CFL 反事实标签工厂（无论信号是否被门控放行，100% 记录），为 AlphaCast 和 MetaLabeler 积累 5K+ 高质量标签。
任务归属交付物关键说明
扩展 Rust 特征至 50 维 Rust core/feature_engine.rs (扩展) 在 Phase 1 的 15 维基础上扩展至 50 维：新增 velocity(6窗口)、跨周期动量(1m/5m/15m/1h)、订单流不平衡 OFI、微观结构熵、Realized Skewness/Kurtosis。

注意力门控融合 Python (PyTorch) models/resnet/feature_fusion.py 经验 Alpha 50d ⊕ ResNet 128d → 注意力门控 → 178d 融合向量。融合延迟 <2ms，与 ONNXRuntime 本地推断对接（Phase 3 迁移至 Triton）。

CFL 反事实标签工厂 Python labeling/counterfactual_labeler.py 无论 G1–G3 是否放行，100% 异步记录：178d 特征快照、AlphaSignal 内容、门控决策、后续 5m/15m/30m/2h 真实收益（多窗口标注）。写入 TimescaleDB cfl_labels 表。V8 新增字段：占位 AlphaCast.conf=0 和 MCTS.value=0，Phase 3/4 填充。
G3 时序百分位门（单标的版） Python (Polars) gating/cross_section.py (单标的) 暂不做跨市场横截面，用 ETH 5m 自身历史分布（滚动 7d 窗口）计算各特征百分位，合成 cs_composite ∈ [0,1]。G3 阈值 cs_composite < 0.35 拒绝，pass_rate 控制在 20–50%。

AlphaCast 模型（正式） Python (PyTorch) models/alphacast/alphacast_model.py
models/alphacast/train_alphacast.py		 Transformer 6L8H，输入 178d×60步，多任务输出：预测收益 ŷ、不确定性 σ、置信度 conf、市场状态。GPU 训练服务器离线训练，导出 TorchScript 推入 Triton 替换 LightGBM 替身。离线 IC 目标 >0.05。

Phase 4 · Tokio MCTS 正式接入 + 全链路串联
Rust UCB1 异步树搜索 · PyO3 Rollout 回调 · <50ms · 无死锁验证
Week 7
▼
目标
用真正的 Tokio 异步 MCTS 替换 Phase 3 的期望收益阈值替身。关键挑战：Python asyncio event loop 与 Tokio runtime 不能在同一线程内嵌套，必须用 spawn_blocking + channel 隔离。
任务归属交付物关键说明
Tokio MCTS 核心 Rust core/mcts_core.rs UCB1 树搜索，4 类动作（buy/sell/hold/close）× 仓位离散化（5档）。8 个 Tokio worker 并发。AlphaCast Rollout 通过 PyO3 回调 Python Triton 客户端（注意：必须在独立 Tokio 线程内 spawn_blocking 调用 Python，禁止在 async 上下文直接调用 PyO3）。Sharpe 加权奖励，γ=0.95，λ=0.3。结果写 Redis（TTL=30s）。
asyncio/Tokio 隔离层 RustPython core/mcts_core.rs (async 边界) 在独立 OS 线程中启动 Tokio runtime（避免嵌套 event loop 死锁）。Python 侧通过 asyncio.get_event_loop().run_in_executor 调用 Rust Future，用 tokio::sync::mpsc channel 传递结果。严格验证：MCTS 结果回传不阻塞 WS 接收线程。
MCTS Worker 调度 Python (asyncio) models/mcts/mcts_worker.py 异步任务池调度，Redis 结果缓存（TTL=30s），降级模式（超时时直接使用 AlphaCast 输出）。超时熔断：MCTS >100ms 自动降级，记录告警。
二次校准整合 Python models/alphacast/alphacast_recalib.py (更新) MCTS 最优路径后接 Temperature Scaling 二次校准。MCTS.path_value 正式填充 CFL 标签的 mcts_value 字段，更新 G4 MetaLabeler 特征集。
pipeline_v2 全链路串联 Python harness/pipeline_v2.py WS 线程 → SHM → 特征 → ResNet → AlphaCast → MCTS (异步) → 门控 → 风控 → 执行。验证：各环节通过 Redis Stream 或 SHM 异步解耦，无阻塞，无死锁。全链路 <220ms（含 MCTS）。
✓ 验收标准
MCTS 1000 次仿真 <50ms；全链路 <220ms（含 MCTS）；MCTS 路径准确率 >55%；WS 接收线程不因 MCTS 阻塞；连续压测 2h 无死锁；Redis MCTS 缓存命中率 >70%
↩ 回滚方案
跳过 MCTS，AlphaCast 直接输出 + 期望收益阈值替代；mcts_worker.py 降级模式固定触发；Rust mcts_core.rs 编译但不调用

5m K线闭合"抢跑"问题
WS 推送闭合事件有网络延迟，错过最佳限价单位置
▼
问题
OKX WS 推送 5m K线闭合事件可能存在几十毫秒网络延迟。如果等收到"闭合"信号再计算特征→推断→下单，会错过最佳限价单位置（价格已经移动）。
解法：本地高精度时钟预计算
Rust 侧维护 NTP 校准的本地时钟，在 5m 周期结束前 200ms，利用已有 Tick 数据提前预计算 99% 的特征。K线一闭合，只做最后的 O(1) 修正（更新 close 价格），立即触发 Alpha 评估。
// core/feature_engine.rs — 5m 预计算策略
fn on_tick(&mut self, tick: &Tick) {
    self.update_partial_bar(tick);
    let remaining_ms = self.time_to_bar_close_ms();
    if remaining_ms < 200 && !self.precomputed {
        // 提前 200ms 预计算所有特征（只缺最终 close 价格）
        self.partial_features = self.compute_50d_features_partial();
        self.precomputed = true;
    }
    if remaining_ms == 0 {
        // K线闭合：O(1) 修正 close 价格，立即触发
        self.partial_features.update_close(tick.price);
        self.trigger_alpha_evaluation();  // <5μs 延迟
        self.precomputed = false;
    }
}

OKX API 限频与 IP 封禁
下单接口 20次/2s，极端行情连续信号触发 IP 短暂封禁
▼
问题
OKX 下单接口限频 20次/2s（私有接口）。极端行情下 Alpha 引擎连续触发信号，可能导致 IP 被短暂封禁（错误码 50011）。同时，规律性的 API 调用节奏容易被 OKX 风控识别为机器人。
解法：令牌桶 + Redis 全局计数器 + 行为伪装
Rust exec_channels.rs 实现令牌桶算法（速率 18次/2s，留 10% 余量）。Redis 维护滑动窗口计数器（INCR + EXPIRE），跨进程共享限频状态。接近阈值时自动降级为挂单或丢弃低置信度信号。BehaviorHumanizer 注入 Poisson 节奏随机化（0.1–0.5s 延迟，±0.002 价格微扰）。
// execution/okx_router.rs — 令牌桶限流
async fn place_order(&self, order: UnifiedOrder) -> Result<Receipt> {
    // 检查 Redis 滑动窗口计数器
    let count = redis_incr_expire("okx:rate:2s", 2).await?;
    if count > 18 && order.confidence < 0.7 {
        // 低置信度信号限频时丢弃（不影响高置信度信号）
        return Err(RateLimit);
    }
    // BehaviorHumanizer：Poisson 延迟 + 价格微扰
    let delay = poisson_delay(0.1, 0.5);
    tokio::time::sleep(delay).await;
    self.sign_and_send(order).await
}

资金费率结算陷阱
每 8h 结算前套利大军平仓，5m 策略误判 regime
▼
问题
ETH 永续每 8 小时结算一次资金费率（UTC 00:00 / 08:00 / 16:00）。结算前约 30–60 分钟，套利大军集中平仓/反向开仓，价格剧烈波动，此时的 5m K线特征失去正常预测性，Alpha 信号质量骤降。
解法：G5 时间门控 + 资金费率方向融合
hard_gating.py 新增 G5 时间门：结算前 30 分钟禁止开新仓（平仓不受影响）。同时将 OKX REST 轮询的实时资金费率（nextFundingRate 字段）作为 Funding Rate Arb 策略的核心输入，把结算期"噪音"转化为可预测的收益机会。
# gating/hard_gating.py — G5 时间门控
FUNDING_TIMES_UTC = [0, 8, 16]  # OKX 每天 3 次结算

def g5_time_gate(signal) -> bool:
    now_utc = datetime.utcnow()
    for h in FUNDING_TIMES_UTC:
        settlement = now_utc.replace(hour=h, minute=0, second=0)
        delta_min = (settlement - now_utc).total_seconds() / 60
        if -5 <= delta_min <= 30:  # 结算前30min到结算后5min
            if signal.action in ['buy', 'sell']:
                return False  # 禁止开新仓
    return True

HMAC 签名与时间戳防重放
OKX 要求 ±30s 时间窗口，服务器时钟漂移导致 50111 错误
▼
问题
OKX REST API 要求请求时间戳与其服务器时间差 <30s（错误码 50111）。VPS 本地时钟漂移、NTP 同步延迟可能导致签名验证失败，尤其在高并发下单时批量失败。
解法：启动时计算 time_offset，所有请求自动补偿
Rust 启动时通过 GET /api/v5/public/time 获取 OKX 服务器时间，计算 time_offset = okx_ts - local_ts。后续所有请求使用 (local_time_ns + time_offset) 作为时间戳。每 5 分钟重新同步一次（防止 VPS 时钟持续漂移）。
// execution/okx_router.rs — 时间戳自动补偿
struct OkxSigner { time_offset_ns: AtomicI64 }

async fn sync_time(&self) {
    let okx_ts = get_okx_server_time().await;
    let offset = okx_ts - SystemTime::now_ns();
    self.time_offset_ns.store(offset, Ordering::Relaxed);
}

fn sign_request(&self, method: &str, path: &str, body: &str) -> Headers {
    let ts = (SystemTime::now_ns() + self.time_offset_ns.load(Relaxed)) / 1_000_000;
    let prehash = format!("{}{}{}{}", ts, method, path, body);
    let sig = hmac_sha256(&self.secret_key, &prehash);  // <5μs
    build_headers(ts, sig)
}

asyncio / Tokio 嵌套死锁
Python asyncio 与 Rust Tokio 在同线程嵌套调用导致死锁
▼
问题
Python asyncio event loop 与 Tokio runtime 在同一线程内不能直接嵌套。如果在 Python async 函数中直接调用 PyO3 暴露的 Rust Future，会触发"Cannot run nested event loops"错误。
解法：独立 OS 线程隔离 + channel 通信
在独立 OS 线程中启动 Tokio runtime，通过 tokio::sync::mpsc channel 与 Python 侧通信。Python 通过 asyncio.get_event_loop().run_in_executor 调用包装函数，完全避免嵌套。MCTS 结果通过 Redis Stream 异步回传，不阻塞 WS 接收线程。
// core/mcts_core.rs — Tokio 独立线程隔离
pub fn spawn_mcts_runtime() -> MctsHandle {
    let (tx, rx) = mpsc::channel::MctsJob(64);
    std::thread::spawn(|| {
        // 独立 OS 线程，完全隔离于 Python asyncio loop
        let rt = Runtime::new().unwrap();
        rt.block_on(async {
            while let Some(job) = rx.recv().await {
                tokio::spawn(run_mcts(job));  // UCB1 树搜索
            }
        });
    });
    MctsHandle { sender: tx }
}
