QTS V8 系统是什么
QTS V8（Quantitative Trading System Version 8）是一套针对 OKX ETH-USDT 永续合约（5分钟周期）的量化交易系统。系统采用 Rust + Python 混合双核架构：

• Rust 侧（你负责）：处理所有延迟敏感的热路径——WebSocket 数据接收、共享内存缓冲、特征计算、订单签名下单、订单状态机、MCTS 树搜索核心。
• Python 侧（Qwen 负责）：处理 AI 模型推断（ResNet/AlphaCast）、策略逻辑（Alpha 引擎）、门控风控、Harness 编排、监控告警、标签积累。

两侧通过三种机制通信：① Protobuf 契约层（schemas/*.proto，所有数据结构的唯一来源）；② SHM 共享内存（Rust 写入，Python 通过 DLPack 零拷贝读取）；③ Redis 温路径（跨进程信号、MCTS 结果缓存）。

你的所有 Rust 模块最终编译为 v8_core_engine.so，Python 侧通过 import v8_core_engine as vce 调用。
开发阶段与你的交付时序
整个项目分 6 个 Phase（Phase 0–5），你主要负责 Phase 0–4 的 Rust 模块交付：

Phase 0（Week 1）：基建期。你交付：Cargo workspace 初始化、prost-build 从 .proto 生成 Rust 结构体、OKX WS 接入（okx_ws.rs）。
Phase 1（Week 2–3）：MVP 最小闭环。你交付：SHM 环形缓冲（buffer.rs）、5m 特征引擎（feature_engine.rs）、OKX 签名下单（exec_channels.rs）、订单状态机（order_state.rs）、PyO3 主入口（lib.rs）。
Phase 2（Week 4–5）：等待 Qwen 的 ResNet 模型，你无新 Rust 任务，但需配合 Qwen 调试 SHM 桥接延迟。
Phase 3（Week 6）：Triton 接入期，你无新 Rust 任务，配合验证 .so 接口。
Phase 4（Week 7）：你交付最复杂模块 mcts_core.rs（Tokio UCB1 树搜索）。

重要：Phase 1 的所有 Rust 模块必须在 Week 3 结束前全部编译通过并提供给 Qwen 测试，否则 Qwen 的 Python 策略层无法运行。

OKX 执行通道 (exec_channels.rs)
HMAC-SHA256 签名 · Token Bucket 限流 · BehaviorHumanizer · 时间戳补偿
Phase 1 · Week 2–3
▼
四个 OKX 专属机制（必须全部实现）
① 时间戳防重放（50111错误）
启动时通过 GET /api/v5/public/time 获取 OKX 服务器时间，计算 time_offset = okx_ts - local_ts 存入 AtomicI64。所有请求时间戳 = SystemTime::now_ns() + time_offset。每 5min 重新同步一次。
② HMAC-SHA256 签名（<5μs）
prehash = timestamp + method.to_uppercase() + request_path + body。使用 ring crate HMAC-SHA256 签名，Base64 编码。Header: OK-ACCESS-KEY, OK-ACCESS-SIGN, OK-ACCESS-TIMESTAMP, OK-ACCESS-PASSPHRASE。
③ Token Bucket 限流（OKX 20次/2s）
Rust 侧实现令牌桶（速率 18次/2s，留 10% 余量）。Redis INCR + EXPIRE(2) 维护滑动窗口计数器。接近阈值时低置信度（conf < 0.7）信号自动丢弃并写告警到 Redis。
④ BehaviorHumanizer（防机器人识别）
下单前注入 Poisson 随机延迟（均值 0.3s，范围 0.1–0.5s）。价格微扰 ±0.002（Limit Order 专属）。通过 tokio::time::sleep 实现，不阻塞其他异步任务。

三个最容易出错的协作边界
1. PyO3 函数名大小写：Python 调用 bridge.get_window()，Rust PyO3 的方法名必须写 "get_window"（不是 getWindow 或 get_Window）。建议直接测试一遍。

2. bytes 传递的 Protobuf 版本：Qwen 的 Python 侧用 grpcio-tools 生成代码序列化 UnifiedOrder，你的 Rust 侧用 prost 反序列化，字段编号必须一致。如果有字段编号分歧，以 schemas/unified_order.proto 文件为准，双方都不得单方面改编号。

3. mcts_core.rs 的 rollout_fn 调用：你在 Rust Tokio 线程内回调 Python callable 时，必须先获取 GIL（Python::with_gil()），调用完成后立即释放。不要在循环内持有 GIL，否则 Qwen 的 asyncio 会被完全阻塞。
执行规则与约束
规则 1：每个 .rs 文件作为独立任务，不合并提交
每次只实现一个文件。不要试图在一次对话中完成所有 Rust 模块，每个文件完成后先验证编译通过，再开始下一个文件。合并 lib.rs 是最后一步。
规则 2：Protobuf 字段编号永不修改
schemas/*.proto 的字段编号（= 1, = 2...）一旦在 Phase 0 确定，任何情况下不得修改。字段名和注释可以改，编号不行。修改编号会导致 Rust 侧和 Python 侧序列化二进制不兼容，调试极难。
规则 3：所有 PyO3 方法必须 allow_threads
凡是有计算或 I/O 的 PyO3 方法，必须用 py.allow_threads(|| {...}) 释放 GIL，否则 Qwen 的 Python 线程会被你的 Rust 计算阻塞。仅构造函数（#[new]）可以不 allow_threads。
规则 4：Redis Key 命名严格遵守约定
所有写入 Redis 的 Key 必须使用"接口契约"页中约定的名称，不得自创新 Key。如果需要新增 Key，先告知 Qwen，双方确认命名后再写入代码。
规则 5：SHM 文件路径固定
共享内存文件路径固定为 /dev/shm/qts_eth5m，不得改变。Qwen 的 shm_bridge.py 会 hardcode 这个路径。进程退出必须 unlink，启动时如果文件存在先 unlink 再重建。
规则 6：Demo 模式与实盘模式通过参数切换
OkxChannel 的构造函数接受 is_demo: bool 参数，Demo 时使用 wss://wspap.okx.com:8443/ws/v5/private（模拟盘），实盘时使用 wss://ws.okx.com:8443/ws/v5/private。切换只改参数，代码逻辑完全相同。
